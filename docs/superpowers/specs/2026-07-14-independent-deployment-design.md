# 地质招聘雷达独立部署设计

## 目标

将现有“地质招聘雷达”从 ChatGPT Sites 迁移为不依赖 ChatGPT 订阅的公开网站。系统使用 GitHub Pages 托管页面，使用 GitHub Actions 每天自动检查公开招聘来源、更新数据并重新发布。用户无需打开 ChatGPT，只需访问固定网址即可查看最新核查结果。

## 已确认约束

- 使用 GitHub 公开仓库，仓库名为 `geology-job-radar`。
- 整套方案保持零费用，不使用付费搜索 API、服务器或数据库。
- 监控对象必须包含原 Word 文件和现有招聘任务中列出的全部集团、子公司、矿区、研究院、技术中心、地勘单位、工程单位与事业单位。
- 用户为地质学硕士，不能把仅限地质工程、资源勘查工程、勘查技术与工程、采矿工程或物探的岗位自动判为匹配。
- 网站是主要查看入口；本阶段不增加邮件、短信或微信推送。
- 免费公开抓取无法保证微信公众号、登录后页面、验证码页面和强反爬页面百分之百成功，网站必须如实显示来源健康状态和覆盖限制。

## 方案选择

采用 GitHub Pages + GitHub Actions：

- GitHub Pages 托管静态招聘看板，不要求用户登录。
- GitHub Actions 每天执行一次抓取、变化检测、专业判定、数据生成、测试与发布。
- 招聘数据、来源登记表、上次抓取指纹和公开运行记录均保存在仓库中，便于追溯。
- 不使用 Cloudflare 凭据、VPS、付费搜索接口或 ChatGPT 定时任务。

## 系统边界

系统由五个独立模块组成：

1. **单位名册**：记录每个被监控单位、所属集团、别名、类别、排名层级和监控优先级。
2. **来源登记表**：记录官方招聘页、集团招聘入口、公开招聘平台页面、允许使用的搜索线索及抓取方式。
3. **监控程序**：抓取公开页面，提取正文和链接，计算标准化页面指纹并识别变化。
4. **招聘判定程序**：识别届别、招聘批次、截止日期、学历、地点、岗位和专业要求，输出匹配等级与判定依据。
5. **静态网站**：读取生成的 JSON 数据，展示有效岗位、临近截止、历史岗位、来源健康度和最后核查时间。

模块通过版本化 JSON 文件通信，抓取、判定和页面展示不互相写入内部状态。

## 单位名册与覆盖规则

`monitor/units.json` 为唯一监控范围清单。原 Word 文件中的组合名称必须拆为独立记录；集团、二级单位、三级单位、矿区、矿井、研究院和基层地勘队分别拥有自己的 `unitId`。

每条记录至少包含：

- `unitId`、单位全称、常用简称和所属集团；
- 行业类别、单位层级、排序等级；
- 官方域名或所属集团招聘入口；
- 可用于发现公告的精确查询词；
- 覆盖状态：直接官方来源、集团入口覆盖、公开平台覆盖或受限来源。

构建测试会校验现有完整名单中的每个名称均能映射到一条单位记录。没有独立招聘页的子单位仍保留独立记录，并通过集团入口、公告附件和精确名称查询进行检查，不因“保底”或弱匹配而删除。

## 来源登记与可信度

`monitor/sources.json` 保存来源，不把搜索结果本身当作官方公告。

来源按可信度分为：

1. 单位官方招聘网站和官网公告；
2. 集团统一招聘入口及其岗位附件；
3. 国务院国资委、人社部、国聘、国家大学生就业服务平台等权威公开平台；
4. 高校就业网和其他公开镜像，仅作为发现线索或备用正文。

每个来源保存 URL、所属单位、可信度、抓取模式、上次成功时间、HTTP 状态、内容指纹和失败次数。官方域名白名单用于判断“官方”标签；镜像链接不能替代官方报名链接。

免费搜索发现仅用于补充没有稳定列表页的单位。搜索发现结果必须经过域名和正文规则复核后才能进入候选列表，不能仅凭标题自动发布为有效岗位。

## 每日数据流

GitHub Actions 按中国时间每天 19:00 左右运行，也支持在仓库中手动触发。

执行顺序：

1. 读取完整单位名册和来源登记表。
2. 并发抓取公开页面，设置超时、重试、速率限制和明确的请求标识。
3. 去除导航、脚本、样式、随机参数和无意义空白，生成稳定正文摘要与 SHA-256 指纹。
4. 与上一次成功指纹比较，仅对新增或发生重要变化的页面继续解析。
5. 提取公告日期、截止时间、批次、岗位、学历、专业、地点、报名链接和实际用人单位。
6. 按专业规则判定“完全匹配、可能匹配、需咨询”以及“专业匹配较弱”。
7. 合并同一单位、同一批次和同一公告的重复记录，并保留多个来源。
8. 将已截止或失效记录移入历史数据，不继续显示为“可投”。
9. 生成网站数据、来源健康摘要和最后核查时间。
10. 运行数据校验、单元测试和生产构建；全部通过后发布 GitHub Pages。

若没有新增岗位，系统仍更新最后核查时间和来源健康状态，但不产生虚假的“新增”记录。

## 专业与招聘匹配规则

匹配以公告专业原文和附件为准：

- 明确包含“地质学”、`0709` 或可确认覆盖地质学的专业目录：`完全匹配`。
- 仅写“地质类”“地学相关专业”且未给出完整目录：`可能匹配`。
- 仅出现地质工程、资源勘查工程、勘查技术与工程、工程地质、物探、采矿工程等工科专业：`需咨询`或`专业匹配较弱`。
- 岗位名称含“地质”但专业要求不包含地质学：不得判为完全匹配。
- 本硕专业一致、院校层次、政治面貌、语言、实习时长等条件单独展示，不被专业标签覆盖。
- 无法可靠解析附件或专业表格时，记录进入待核验区，不自动进入“必投”。

每条判定必须保存命中的专业原文和规则编号，网站可直接显示判定依据。

## 网站功能

现有页面视觉和筛选能力继续保留，并增加：

- 当前有效、临近截止、今日新增、历史岗位四种状态；
- 来源健康度：成功、失败、受限、尚未配置；
- 监控单位覆盖统计和可搜索的单位名册；
- 最近一次成功核查时间和最近一次完整发布版本；
- 每条岗位的首次发现时间、最后确认时间与变化摘要；
- 失效岗位历史页，防止旧链接继续显示为可投；
- 明确区分官方公告、官方报名、权威平台和非官方镜像。

页面默认只展示仍可投的岗位，筛选逻辑在浏览器本地运行，不收集用户信息。

## 异常处理

- 单个来源超时或返回错误时重试，其他来源继续处理。
- 抓取失败时保留该来源最后一次成功数据，并标记为“检查失败”，不得把岗位批量删除。
- 连续失败达到阈值后在网站来源状态区突出显示，方便后续更换 URL 或抓取方式。
- 页面结构变化导致解析字段缺失时，候选记录进入待核验区，不覆盖上一次已验证数据。
- 数据校验或构建失败时不发布，线上网站继续保留上一个成功版本。
- 动态渲染、登录或验证码来源标记为“受限”，并通过集团入口、权威平台和公开搜索线索进行补充。

## GitHub 工作流与发布

仓库包含两个工作流：

- `monitor.yml`：每日定时及手动运行，更新招聘数据和来源状态；仅当生成数据发生变化时提交结果。
- `deploy.yml`：在主分支网站或数据变化后构建静态页面并发布 GitHub Pages。

生产网址使用 GitHub Pages 项目路径。构建配置必须正确处理 `/geology-job-radar/` 基础路径，所有静态资源和站内链接不得依赖根域名。

仓库不保存密码、Cookie、个人简历或 API 密钥。公开监控仅请求无需登录即可访问的页面。

## 测试与验收

自动测试至少覆盖：

- 完整单位清单无遗漏、无重复 ID、组合单位已拆分；
- 来源 URL 格式、单位映射和官方域名标签一致；
- 页面指纹标准化、变化检测和重复公告合并；
- 2027届、应届硕士、截止日期和专业匹配规则；
- 仅限地质工程的岗位不会被判为地质学完全匹配；
- 抓取失败时保留旧数据，截止岗位正确进入历史；
- 网站搜索、筛选、基础路径、生产构建与静态资源链接；
- 360px 手机宽度无横向溢出，键盘操作可用。

验收条件：

1. 独立 GitHub Pages 网址可公开访问。
2. GitHub Actions 可手动成功运行，并配置每日自动核查。
3. 网站显示本次核查时间、有效岗位和来源健康度。
4. 现有三条种子招聘记录完整迁移，专业判断不退化。
5. 原完整名单中的每个单位均出现在覆盖名册中。
6. 停用 ChatGPT 任务后，独立网站和 GitHub 定时工作流仍可运行。

## 不在本阶段范围

- 绕过登录、验证码、访问控制或网站反爬限制；
- 付费搜索 API、付费服务器、短信或商业邮件服务；
- 自动填写或提交网申；
- 将非官方搜索线索伪装为已验证的官方招聘；
- 承诺所有微信公众号或未被搜索引擎收录的公告均能实时发现。
